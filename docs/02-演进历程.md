# 02 · 演进历程：从 PowerShell 脚本到 Tauri 应用

本页按时间线记录整个迭代过程，便于回顾“为什么是现在这个样子”。

## 阶段一：PowerShell + WinForms 单脚本

最初版本是一个 `SmartCopyTool.ps1`（WinForms 界面）+ 一个 `.cmd` 启动器，
核心是把界面参数拼成 `robocopy` 命令行执行。保存在 [`legacy-powershell/`](../legacy-powershell)。

特点：零安装、改动小；但 UI 是 WinForms 老控件，且踩到了 PowerShell 的并发坑。

## 阶段二：修复三个实际问题

### 问题 1 — 复制粒度（内容 vs 文件夹）

`robocopy <源> <目标>` 会把**源的内容**复制进目标。要得到“目标下出现一个同名文件夹”，
必须把目标改写为 `目标\源末级名`。

**修复**：新增“在目标下创建源同名文件夹”勾选项（默认开），开启时自动把
`D:\work_files` 改写为 `D:\work_files\desktop`。

### 问题 2 — 多任务只跑一个就停

原脚本用 `Process.add_Exited` 事件来串接队列里的下一个任务。该事件在 .NET 线程池的
**后台线程**触发，在 PowerShell 的 STA WinForms 运行空间里执行事件脚本块极不稳定，
经常静默死掉——表现为“跑完第一个就不动了”。

**修复**：改用跑在 **UI 线程**上的 `System.Windows.Forms.Timer`（定时轮询）来驱动队列：
检测当前进程是否结束 → 记录结果 → 启动下一个。全程在 UI 线程，消除跨线程重入。

### 问题 3 — 跑完窗口“消失”

是问题 2 崩溃的连带现象。队列改为 Timer 驱动后，整批结束会正常打印汇总、状态回到“就绪”，
窗口保留不自动关闭。

### 顺带：打包成 exe

- 用系统自带的 `csc.exe`（.NET Framework 4.8）编译了一个 4.6KB 的**启动器** `Launcher.cs` →
  `SmartCopyTool.exe`，双击即以无控制台窗口方式拉起 `.ps1`。无需联网、无第三方依赖。
- 另提供 `Build-Exe.ps1`，用 `ps2exe` 把脚本**整体**打包成单文件 exe（首次需联网装模块）。

> 踩坑记录：用 Write 工具生成的 `.ps1` 默认无 BOM，而 PowerShell 5.1 会按系统 ANSI 代码页
> 读取无 BOM 文件，导致中文乱码、解析报错。解决办法是给脚本文件加 UTF-8 BOM。

## 阶段三：重做成正式项目（当前主线）

为了把 6 项增强需求做扎实、并把界面做现代化，决定引入开发框架重写。

### 技术选型

对比了 WPF/.NET、Electron、Tauri、保持 PowerShell 四个方向后选择 **Tauri**：

- 原生小体积 exe（约 5–15MB 量级），远小于 Electron 的百 MB 级。
- 前端用 Web 技术栈（React + TS + Tailwind），UI 上限高、开发体验好。
- 后端用 Rust，直接、可靠地调度 `robocopy` 进程与并发。

代价：需要一次性安装 Rust 工具链 + MSVC C++ Build Tools + WebView2（见
[05 · 开发与构建指南](./05-开发构建指南.md)）。

### 本机工具链现状（决策依据）

迁移时机器上：有 Node 24 / npm 11 / winget / git；**没有** dotnet SDK、MSBuild/VS、Rust。
因此用官方脚手架 `create-tauri-app`（React-TS 模板）生成骨架，前端依赖可直接 `npm install`
并构建验证；Rust 侧代码完成编写，编译留待安装工具链后进行。

### 成果

- 任务级参数 + 基于“N 个 worker 共享队列”的有限并行。
- 预扫描总量 + 解析 robocopy 输出，得到实时进度 / 速度 / ETA。
- 复制后校验（quick 比大小 / full 比 sha256），与系统语言无关。
- 拖动排序、弹窗编辑、命名预设持久化、完成通知 + 提示音。
